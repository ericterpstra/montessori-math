# PRD 17 — Install-to-Tablet PWA: full offline, zero dependencies

**Status:** Not started
**Effort:** S — no UI work, four small files touched plus one ~80-line build script; the only care point is the service-worker fetch handler, which is fully specified below.
**Depends on:** nothing (wave 1 is complete; this wraps the existing static build)

## Why

A family's math tablet should work at the kitchen table, in the car, or at a cabin with no Wi-Fi. This PRD makes the site installable to an Android tablet's home screen (icon, standalone window, no browser chrome) and makes every material, lesson, worksheet builder, and guide load with the network fully off after one online visit. Parents get an app-like experience for the child without an app store, an account, or a single new dependency; the child gets the golden beads even when the internet is down.

The site is already fully client-side (seeded RNG worksheets, pure-model materials, no runtime network requests), so offline is purely a caching problem: a web app manifest + a hand-rolled service worker generated at build time.

## Binding product rules

From `CLAUDE.md` — restated here because the implementer must not drift:

1. **No new npm dependencies (rule 5).** `vite-plugin-pwa`, `workbox-*`, and any icon/image package are **forbidden**. The service worker is generated by a plain Node script (`node:fs`, `node:path`, `node:crypto` only). Runtime deps stay exactly `react`, `react-dom`, `react-router-dom`; devDependencies unchanged.
2. **Fully static and offline-friendly (rule 4).** This feature *implements* that rule. The service worker must only ever cache and serve same-origin files that are already in `dist/`. No external requests of any kind.
3. **No tracking, no analytics, no localStorage progress (rule 1).** The Cache API storage added here holds *site assets only* — never user data, never usage records. No install-nag banners, no "you've been offline N days" messaging, no gamification of any kind.
4. **Print stays first-class (rule 3).** Offline mode must not degrade printing: all print CSS is bundled, so worksheets print (color and `.bw`) with the network off. Nothing in this PRD touches `print.css`, `.print-sheet`, or `.sheet-page`.
5. **No UI changes.** No install-prompt buttons, no "offline ready" toasts, no update notifications. The feature is invisible chrome; the browser's own install affordance is the whole UX.
6. **TypeScript strict + verbatimModuleSyntax** applies to the one `src/` change (`src/main.tsx`). The build script `scripts/generate-sw.mjs` is plain ESM JavaScript outside `tsconfig.json`'s `include` (`["src", "vite.config.ts"]`) and is not type-checked — that is intentional and fine.

## Design decisions (locked — do not revisit)

- **Manifest** at `public/manifest.webmanifest`: `name`/`short_name` `"Montessori Math"`, `start_url: "/"`, `scope: "/"`, `display: "standalone"`, `background_color`/`theme_color` `"#faf7f0"` (this is the site's `--paper` token, `src/styles/tokens.css` line 42 — the hex literal is required here because a manifest cannot read CSS variables). Two SVG icons: the existing `public/favicon.svg` (`purpose: "any"`) and a **new** `public/icon-maskable.svg` (`purpose: "maskable"`) — the same golden bead inset to 60% inside a `#faf7f0` rounded square so platform mask cropping never clips the bead.
- **Honest icon limitation:** SVG manifest icons work on Chrome/Edge/Android — the actual target (home tablets). iOS Safari ignores SVG manifest icons; we add `<link rel="apple-touch-icon" href="/favicon.svg">` anyway, and accept that an iOS home-screen tile will show a page screenshot. Generating PNG icons is **out of scope** (would need an image dependency, violating rule 5).
- **Service worker is generated, not hand-maintained:** `scripts/generate-sw.mjs` (new) runs as the last step of `npm run build`, walks `dist/`, and writes `dist/sw.js` with a complete precache list. `dist/sw.js` is a build artifact — never committed, never edited by hand.
- **Cache versioning:** `CACHE = 'mm-' + VERSION` where `VERSION` = first 12 hex chars of the SHA-256 of the sorted asset-path list joined with each file's byte size. Any build that changes any output file changes the version; identical builds produce byte-identical `sw.js` (deterministic).
- **Fetch strategy:** cache-first. GET + same-origin only. Navigations (`request.mode === 'navigate'`) are answered from the cached `'/'` (the SPA shell — React Router then resolves `/materials/golden-beads` etc. client-side), falling back to network. Everything else: cache match, else network with successful responses copied into the cache, with a final catch returning the cached `'/'`.
- **Lifecycle:** `skipWaiting()` on install, delete all non-current `mm-*` caches + `clients.claim()` on activate. Updates are silent: a new build's `sw.js` differs byte-wise, Chrome installs it on the next visit, and the next reload serves the new assets.
- **Registration** in `src/main.tsx`, production-only (`import.meta.env.PROD` — typed via the existing `"types": ["vite/client"]` in `tsconfig.json`), so `npm run dev` is completely unaffected (`sw.js` only exists in `dist/`).
- **Secure-context reality (document, don't fight):** service workers require HTTPS **or** `localhost`. `http://localhost:4173` (via `npm run preview`) gets the full offline experience on the dev machine. Plain `http://192.168.1.208:4173` on the LAN will load the site but will **not** register the worker or offer install. For tablet testing, use Chrome's developer flag `chrome://flags/#unsafely-treat-insecure-origin-as-secure` (documented in the README section, step 12 of QA). Setting up LAN HTTPS is out of scope.
- **Docs:** README gains an "Offline & install" section (spec below), including the developer escape hatch for a stale worker (DevTools → Application → Service workers → Unregister). No changes to `plan/README.md` beyond the PRD table row.
- **No vitest for the script** — it is build tooling operating on `dist/`, outside the vitest scope (`src/**/*.test.{ts,tsx}`, node env). Verification is via deterministic build assertions (see Testing).

Notes for the implementer:
- The gradient stops and bead geometry are copied verbatim from `public/favicon.svg` — do not restyle the bead.
- At 60% scale the bead's visual radius is 7.8 of 16 (48.75% of the half-width), comfortably inside the W3C maskable safe zone (a centered circle of 40% radius must contain critical content; the bead center region does, and the background is uniform out to the edges).

- The `rx="4"` corner rounding only shows against the rare launcher mask that reaches the extreme corners; standard circle/squircle masks crop inside it. If a launcher ever shows corner slivers, flattening to `rx="0"` is an acceptable follow-up tweak — do not do it preemptively.

### Step 4 — `scripts/generate-sw.mjs` (new; create the `scripts/` directory)

The whole script — copy verbatim. Node ≥ 20 (repo README already requires 20+; dev machine runs 24), imports from `node:` builtins only.

```js
#!/usr/bin/env node
/**
 * generate-sw.mjs — generates dist/sw.js, a precaching service worker listing
 * every file in dist/. Runs as the final step of `npm run build`.
 * No dependencies: node:fs, node:path, node:crypto only. Do not add packages.
 */
import fs from 'node:fs'
import path from 'node:path'
import crypto from 'node:crypto'
import { fileURLToPath } from 'node:url'

const distDir = path.resolve(path.dirname(fileURLToPath(import.meta.url)), '..', 'dist')

if (!fs.existsSync(path.join(distDir, 'index.html'))) {
  console.error('generate-sw: dist/index.html not found — run `vite build` first.')
  process.exit(1)
}

/** Recursively list every file under dir; returns absolute paths. */
function walk(dir) {
  const files = []
  for (const entry of fs.readdirSync(dir, { withFileTypes: true })) {
    const full = path.join(dir, entry.name)
    if (entry.isDirectory()) files.push(...walk(full))
    else if (entry.isFile()) files.push(full)
  }
  return files
}

const entries = walk(distDir)
  .filter((file) => path.relative(distDir, file) !== 'sw.js') // never precache the worker itself
  .map((file) => {
    const rel = path.relative(distDir, file).split(path.sep).join('/')
    return {
      url: rel === 'index.html' ? '/' : '/' + rel, // the SPA shell caches under '/'
      size: fs.statSync(file).size,
    }
  })
  .sort((a, b) => (a.url < b.url ? -1 : 1))

const hash = crypto.createHash('sha256')
hash.update(entries.map((e) => `${e.url}:${e.size}`).join('\n'))
const VERSION = hash.digest('hex').slice(0, 12)

// One asset per line, two-space indent, double-quoted — the QA parity check
// (`grep -c '^  "' dist/sw.js`) depends on this exact formatting.
const assetLines = entries.map((e) => `  ${JSON.stringify(e.url)},`).join('\n')

const sw = `/* Generated by scripts/generate-sw.mjs — do not edit, do not commit. */
const CACHE = "mm-${VERSION}";
const ASSETS = [
${assetLines}
];

self.addEventListener("install", (event) => {
  event.waitUntil(
    caches
      .open(CACHE)
      .then((cache) => cache.addAll(ASSETS))
      .then(() => self.skipWaiting())
  );
});

self.addEventListener("activate", (event) => {
  event.waitUntil(
    caches
      .keys()
      .then((keys) => Promise.all(keys.filter((key) => key !== CACHE).map((key) => caches.delete(key))))
      .then(() => self.clients.claim())
  );
});

self.addEventListener("fetch", (event) => {
  const request = event.request;
  if (request.method !== "GET") return;
  const url = new URL(request.url);
  if (url.origin !== self.location.origin) return;

  if (request.mode === "navigate") {
    // SPA: every navigation is answered by the cached shell; React Router
    // resolves the actual route (/materials/golden-beads, etc.) client-side.
    event.respondWith(caches.match("/").then((cached) => cached || fetch(request)));
    return;
  }

  event.respondWith(
    caches
      .match(request)
      .then(
        (cached) =>
          cached ||
          fetch(request).then((response) => {
            if (response.ok) {
              const clone = response.clone();
              caches.open(CACHE).then((cache) => cache.put(request, clone));
            }
            return response;
          })
      )
      .catch(() => caches.match("/"))
  );
});
`

fs.writeFileSync(path.join(distDir, 'sw.js'), sw)
console.log(`generate-sw: wrote dist/sw.js (cache mm-${VERSION}, ${entries.length} assets)`)
```

Implementation notes (already encoded above — listed so nobody "fixes" them):
- The embedded worker source contains **no** backticks and **no** `${` sequences of its own, so a single outer template literal with two interpolation points (`VERSION`, `assetLines`) is safe. Keep it that way if you edit the template.
- `dist/index.html` maps to `'/'` — both the precache entry and the navigate handler use `'/'`, and Vite's preview server (and any static host) serves `index.html` at `'/'`, so `cache.addAll` stores it under the URL the handler looks up.
- Sorting by URL + skipping `sw.js` makes reruns deterministic and idempotent (running the script twice over the same `dist/` yields byte-identical output).
- `cache.put` of the response clone is deliberately not awaited — it is opportunistic best-effort top-up; the response returns to the page immediately.
- The final `.catch(() => caches.match("/"))` is a last-resort so an offline request for an uncached URL still renders the app shell instead of a browser error page.

**Check:** run `npx vite build && node scripts/generate-sw.mjs`. Console prints `generate-sw: wrote dist/sw.js (cache mm-<12 hex chars>, N assets)`.
```ts
// Offline support: precaching service worker, production build only
// (dist/sw.js is generated by scripts/generate-sw.mjs; it does not exist in dev).
if ('serviceWorker' in navigator && import.meta.env.PROD) {
  window.addEventListener('load', () => {
    navigator.serviceWorker.register('/sw.js').catch(() => {
      // Registration is best-effort: on http:// LAN origins (no secure context)
      // or older browsers the site simply stays online-only.
    })
  })
}
```

## Testing

The existing suite must stay green, and the script is verified by the following **named build assertions**, run from the repo root. Each has a concrete expected output — treat them like test cases.

**BA-1 `sw-deterministic`** — same sources ⇒ byte-identical worker:
```bash
npm run build && cp dist/sw.js /tmp/sw-run1.js && npm run build && diff /tmp/sw-run1.js dist/sw.js && echo PASS
```
Expected: `PASS` (diff prints nothing). This proves the sorted walk + size-based hash is deterministic.

**BA-2 `sw-version-changes-on-content-change`** — a real content edit ⇒ new `VERSION`:
```bash
grep -o 'mm-[0-9a-f]\{12\}' dist/sw.js | head -1        # note value A
# append a visible character to any user-facing string in src/App.tsx (a plain
# mtime-only `touch` is NOT enough — Vite hashes content, not timestamps)
npm run build
grep -o 'mm-[0-9a-f]\{12\}' dist/sw.js | head -1        # value B
# revert the edit, rebuild, confirm the version returns to value A
```
Expected: A ≠ B; after revert + rebuild, version equals A again.

**BA-3 `sw-precache-completeness`** — every file in `dist/` appears in `ASSETS` (the `index.html → "/"` mapping and the `sw.js` self-exclusion make the counts line up one-to-one):

## Out of scope

- **PNG icon generation** (any size): would require an image-processing dependency — forbidden. SVG-only icons, with the iOS tile limitation accepted and documented.
- **`vite-plugin-pwa` / Workbox / any npm package** — explicitly forbidden by the locked design and hard rule 5.
- **Update-notification UI** ("new version available — reload"): updates stay silent by design; no UI chrome.
- **Install-prompt UI** (`beforeinstallprompt` handling, custom install buttons): the browser's native affordance is the whole install UX.
- **HTTPS on the LAN** (mkcert, reverse proxies, tailscale serve): the `chrome://flags` workaround is documented for tablet testing; real HTTPS arrives with public deployment (out of scope per CLAUDE.md).
- **Offline analytics/telemetry of any kind** — permanently out of scope (hard rule 1).
- **Background sync, push notifications, periodic background fetch** — no use case; the site has no server.
- **Runtime caching of cross-origin resources** — none exist (hard rule 4) and the worker deliberately ignores non-same-origin requests.
